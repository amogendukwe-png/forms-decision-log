Shared lookup data for common dropdowns (KV store / lookup table) 
This one is directly about how you implement and design form controls for common lists (parishes, countries, etc.), and how you keep them consistent.
* Date: 2 Apr 2026
* Area: Forms platform / data model for dropdown options
* Question / problem: How should we store and provide reusable lists like parishes, countries, titles (Mr/Miss/Mrs), schools, etc., so forms don’t hardcode options?
* Decision:
   * Use a central store (conceptually a KV store / lookup table) to hold reusable dropdown/select lists such as parishes, countries, titles, schools, and similar constants.
   * Implement this as a lookup table in the database migrated “like everything else”, plus a small API endpoint so front‑end forms can fetch these lists at runtime instead of hardcoding them.
   * This fits into the existing infrastructure; no new infra changes are needed.
* Why (rationale, inferred):
   * Ensure consistency in option lists across forms (e.g. the same set of parishes everywhere).
   * Make it easier to maintain and update options centrally rather than editing many separate forms.
   * Keep the architecture simple by reusing existing migration and API patterns.
* Source:

   * Laron’s clarification and confirmation that this would be implemented as lookup tables plus an API, with no infra changes. ​⁠ ​⁠ ​⁠
